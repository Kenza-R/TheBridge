01

Product Overview & Core Philosophy

What this product is, what it is not, and the non-negotiable design principles.
One-Sentence Definition

Bridge detects physician burnout before it becomes crisis — using EHR workflow signals and wearable biometrics — then routes clinicians to peer support through a privacy-first architecture that is entirely controlled by the physician, never the employer.

What Bridge is
  • An ambient signal detector for distress
  • A physician-controlled peer support launcher
  • A safe, low-friction decompression space
  • An anonymized aggregate health dashboard for hospital leadership
  • A privacy-first trust architecture
What Bridge is not
  • Not a surveillance tool for HR or administration
  • Not a crisis hotline (that's a referral partner)
  • Not a mandatory wellness check-in
  • Not a replacement for professional mental health care
  • Not something that shares individual data with employers
Core design principles
  • Physician sovereignty — every control belongs to them
  • No auto-send — humans approve every outreach
  • Employer firewall — aggregate-only hospital view
  • Friction minimalism — max 2 taps to any feature
  • Non-clinical tone — language that doesn't pathologize
Seven-Step Product Flow
1

Signal Detection

EHR workflow data (Epic FHIR), wearable biometrics (Apple Watch, Garmin), and optional self-report tags are continuously scored against the burnout model. Score is computed locally on-device and never sent to employers.

2

Threshold Trigger

When the composite distress score crosses a configurable threshold (default: 70/100 for 3 consecutive days, or 90/100 on a single shift), a soft in-app notification appears. No push notification, no sound — just a subtle banner on next app open.

3

Three-Option Interface

The physician sees three options — not a wall of text: Reset Now (90-second micro-reset, solo), Talk to Someone (Circle of Trust peer launcher), or Handle Admin (dismiss + schedule a check-in). All three are equally weighted — no "emergency" visual hierarchy that creates alarm.

4

Physician Decision

The physician chooses their path. Bridge never escalates automatically. No auto-notify, no auto-escalate. The physician is always the actor.

5

Circle of Trust Approval Gate

If the physician selects "Talk to Someone," they see their pre-configured Circle of Trust contacts. They choose who to notify. They preview the message before it sends. A single "Send" tap dispatches the request.

6

Peer Response

The selected circle member receives a private in-app message (not SMS, not email — app-only). They can respond, schedule a call, or decline with a supportive template. The hospital never sees this exchange.

7

Recovery & Loop Closure

48 hours post-trigger, Bridge gently asks (one optional question): "How are you feeling today?" Score recalibrates. If score stays elevated for 7+ days, Bridge surfaces the professional referral pathway — EAP, licensed counselor, crisis line — as a gentle suggestion, never a mandate.

Signal Detection Engine

How Bridge knows something is wrong — three input layers, one composite score.
Layer 1 — EHR Workflow Signals (Epic)

Sourced via Epic FHIR R4 read-only scopes. No PHI. Only metadata about clinician behavior patterns.

20%

12%

Layer 2 — Wearable Biometrics (opt-in)

Physician explicitly connects device. Data stored only on-device + encrypted user-private vault. Never shared with hospital.

Layer 3 — Self-Report Tags (in-app, optional)

One-tap emotional tags available from the home screen. Never mandatory. When used, they boost signal confidence significantly because they represent conscious self-disclosure.

Rough shift Lost a patient Bad handoff Overwhelmed Running on empty Need a break Actually fine today Family stuff Conflict with colleague Understaffed
Scoring Architecture

Each layer generates a normalized sub-score (0–100). The composite distress score is: (EHR × 0.5) + (Wearable × 0.35) + (Self-report × 0.15) if wearable is connected; or (EHR × 0.80) + (Self-report × 0.20) if wearable is not connected. The model is personalized — scores are computed against the individual's own 30-day baseline, not population averages. Day 1–30 is a silent calibration period with no triggers.

Hard Triggers (immediate, single-event)

Some events bypass the scoring model and trigger the three-option interface immediately: (1) Documenting a patient death in Epic where physician was primary, (2) Self-report tag "Lost a patient," (3) Wearable detects HRV crash below absolute threshold (not just relative) combined with sleep <4h. These are treated as "known-bad" events, not statistical outliers.

03

Circle of Trust

Physician-owned peer network. No employer visibility. Full physician control at every step.
What is the Circle of Trust?

The Circle of Trust is a private list of 2–6 trusted colleagues that each physician builds during onboarding. These are the people they want to be able to reach when things get hard — or who they want notified that they had a rough day. No one outside the physician's selected circle can see its membership. The hospital cannot see who is in whose circle. The circle is stored encrypted in a physician-private vault.

Onboarding: Building the Circle (Step-by-Step)
1

Invite via internal directory

Search colleagues by name or specialty. Only existing Bridge users appear. No cold outreach — both parties must have accounts.

2

Colleague accepts (opt-in)

Receiving physician gets a private in-app invite. They can accept, decline, or "accept with limits" (read-only supporter, not a responder).

3

Assign roles

For each circle member: First Reach (always notified first), Backup (if First Reach doesn't respond in 2h), or Check-In Buddy (scheduled regular touchpoints).

4

Set notification preferences

Per circle member: "Always notify," "Notify only if I haven't responded in 4h," or "Never auto-suggest — only if I manually select."

5

Set threshold for circle notification

Should the app suggest notifying a circle member after a hard trigger? Physician sets this: "Always ask me," "Suggest after 2 hard days in a row," or "Never suggest — I'll initiate."

How the Circle Works (Live Flow)

When the physician hits the "Talk to Someone" option in the three-option interface:

  • Circle members appear as cards: name, role, specialty, availability status
  • Availability status comes from Epic (in surgery = unavailable, off shift = green)
  • Physician selects 1 or more members to notify
  • Default message appears: "I had a rough one. Could use a quick chat when you're free." — fully editable
  • Physician previews the message before it sends — full-screen review, no accidental dispatch
  • One tap: Send. Message goes in-app, encrypted, to selected circle members only
  • Receiving member gets a private in-app notification with a single CTA: "Respond to [Name]"
  • No email, no SMS, no calendar invite auto-created — everything stays in-app by default
  • Physician can optionally "Request a walk together" or "Schedule 15 min to talk" as response options
Anti-Surveillance Design

The hospital admin dashboard shows only: "X% of physicians used peer support this month." No names. No who-contacted-whom. No message content. Not even which department — aggregate institution-wide only below a minimum group size of 10.

Third-Party Concern Flagging — "Someone on My Radar"

A physician can flag a colleague they're worried about — without the colleague knowing who flagged them. This is separate from the Circle of Trust flow.

How it works
  • Physician goes to "Team" tab → selects a colleague
  • Taps "I'm a bit worried about them"
  • Optional: adds a private note (visible only to their Circle Admin, never to employer)
  • Bridge records the flag internally. If that colleague's own score is also elevated, Bridge surfaces the peer support option more prominently in the colleague's next session
  • The concerned physician is never identified to the flagged colleague
  • No escalation — this is a soft signal only
Privacy logic
  • Flags are visible only to the platform's Care Coordination module (not hospital admin)
  • Multiple flags on the same person from different colleagues → Bridge suggests a direct check-in, not an HR alert
  • Flags auto-expire after 14 days if no corroborating signals
  • A physician can see if they have been "flagged" — shown as "A colleague is thinking of you" — but never who flagged them
04

Peer Support Network — POST / RISE Models

Structured peer support inspired by Johns Hopkins RISE program and Stanford WellMD POST model.
RISE Model (Johns Hopkins)

Resilience In Stressful Events. Trained peer responders — physicians who volunteered for 4-hour Bridge peer training — available for immediate same-shift response after adverse events.

  • RISE responders appear in "Peer Support" tab, not Circle of Trust
  • Anonymous request — physician doesn't know which RISE responder comes
  • Response SLA: within 30 minutes during business hours
  • For critical events (patient death, near-miss): auto-suggested within 2h of documentation
  • RISE session is entirely confidential — not documented in any HR system
  • RISE responders log session metadata only (time, duration, no content) for program reporting
POST Model (Kaiser Permanente)

Peer Organizational Support Team. Ongoing peer coaching relationship, not just crisis response. Scheduled regular check-ins, specialty-matched.

  • POST coaches are senior physicians, 1+ year with Bridge
  • Matching criteria: same specialty, similar career stage, opt-in availability
  • Frequency: biweekly 20-min check-in (via in-app video or voice)
  • Not therapy — explicit framing: "This is a colleague conversation, not a clinical session"
  • POST coach can refer to EAP or licensed counselor if needed, with physician consent
  • POST relationship shows in the physician's profile as "Peer Coach" — visible only to them
Social Recovery Scheduling

One underrated resilience mechanism is structured informal time with colleagues — walks, meals, collegial activities. Bridge makes this trivially easy to schedule.

Walk Together
Quick 15-min walk invite sent to circle members. Pull from shared availability (Epic calendar read). One tap to accept.
Lunch Sync
Finds overlapping 30-min windows in the next 3 days. Shows as "Dr. Chen is free Tuesday 12:15." One tap to confirm.
Coffee Debrief
For post-adverse-event processing. Structured agenda suggestion: "What happened, what you felt, what you'd do differently." No documentation.
05

Mental Health Space

In-app decompression tools — solo micro-resets, guided breathing, meditation, and structured debriefs.
Design Principle

Every tool in the Mental Health Space is available in 90 seconds or less. Physicians won't use anything that requires 10 minutes. The barrier is the duration. Every tool should feel like a coffee break, not a therapy session. Animations use calming colors (not clinical white or warning red). Tone is warm, not instructional.

90-Second Micro-Reset

The fastest intervention. Available directly from the three-option interface ("Reset Now") and from the home screen.

  • Box Breathing (4-4-4-4) — Animated visual guide. Inhale 4s, hold 4s, exhale 4s, hold 4s. 4 cycles = 64 seconds.
  • 4-7-8 Breath — For acute anxiety. Inhale 4s, hold 7s, exhale 8s. 3 cycles = ~57 seconds.
  • Extended Exhale — For nervous system downregulation. Inhale 4s, exhale 8s. 6 cycles = 72 seconds.
  • Visual: full-screen ambient color fill (blue-green gradient) that expands/contracts with breath
  • Haptic feedback on Apple Watch if connected
  • After session: "How do you feel?" 1–5 scale (optional)
3-Minute Guided Debrief

Structured emotional processing after difficult events. Voice-guided or text-guided. Evidence base: Pennebaker expressive writing + structured debriefs from RISE program.

  • Prompt 1 (60s): "What happened in that room?" — describe the clinical event (stays private, never transmitted)
  • Prompt 2 (60s): "What did you feel in that moment?" — name the emotion without judgment
  • Prompt 3 (60s): "What would you tell a colleague in that same situation?" — reframe with compassion
  • All text stored locally on device. Not synced to server. Physician can export or delete at any time.
  • After debrief: option to share summary (not content) with Circle of Trust member: "I worked through something hard today. I'm okay."
Meditation Library

10 curated audio tracks, each under 5 minutes. Designed specifically for healthcare workers (not generic Calm/Headspace content).

  • "After the code" — for post-resuscitation emotional processing (4 min)
  • "End of shift exhale" — transition from clinical to personal (3 min)
  • "Between patients" — micro-reset for high-volume clinic days (2 min)
  • "Night shift grounding" — for 3am when everything feels harder (5 min)
  • "Imposter moment" — for the weight of uncertainty in complex cases (4 min)
  • Hosted as offline audio — no streaming latency in ICU with poor wifi
Progress Visualization (Private)

A private wellbeing timeline visible only to the physician. Never shared with hospital. Helps them see patterns over time.

  • Chart: distress score over 90 days (smooth line, no alarming red)
  • Overlaid events: adverse outcomes (marked as dots), peer support sessions (marked), wearable dips
  • Correlation insights: "Your score tends to rise in weeks with 3+ night shifts" — surfaced after 60 days of data
  • Monthly insight card: "This month you used peer support twice. Last month: zero. You're building the habit."

Notification System

Exact copy, trigger conditions, and style guidelines for every notification type.
Notification Design Rules

No clinical language. No alarm words ("crisis," "emergency," "alert"). No red badges. No sound during patient care hours (default silent 7am–7pm). Every notification has a single clear action. Maximum 1 notification per day per physician. The notification is an invitation, not a directive.

Privacy Architecture for Wearable Data
  • iOS: data sourced from HealthKit. Android: Health Connect API.
  • Raw biometric values never leave the device
  • Only derived signals (normalized score deltas) are sent to Bridge backend — never raw HRV numbers
  • Physician can revoke wearable access at any time without affecting Circle of Trust
  • If hospital purchased Bridge: hospital can see only "X% of physicians have wearables connected" — no individual data
  • Wearable data is never used for insurance, credentialing, or performance review — enforced contractually in all hospital agreements
Apple Watch Haptic Integration

For physicians wearing Apple Watch: breathing exercises deliver synchronized haptic taps (tap on inhale start, double-tap on exhale start). End of breathing session: gentle "success" haptic. RISE peer responder requests: single pulse to wrist (no sound, no visual disruption during patient care).

08

Hospital Administration Dashboard

Anonymized, aggregate-only. Hospital leadership sees trends, never individuals.
Hard Privacy Rules (Contractual)

These are enforced in the hospital contract and technically enforced in the data model: (1) No individual physician data visible to HR, CMO, or department heads. (2) No department-level breakdown unless that department has 10+ active physicians. (3) No correlation between distress scores and performance reviews — any such request is rejected at the API level. (4) Audit logs available to privacy officer showing all dashboard queries — physicians have right to request these logs.

Dashboard Panels

Overall Wellbeing Index
admin/dashboard/overview
Shown to: CMO, Chief Wellness Officer
Metric: % of active physicians in "elevated distress" zone this week vs. 30-day rolling avg. Trend line only — no absolute number if <10 total active.
Peer support utilization rate: % who initiated a peer support session this month. Compares vs. national benchmark.
Micro-reset usage: Total breathing/meditation sessions (not by who, just total count institution-wide).
Temporal Pattern View
admin/dashboard/patterns
Shown to: Chief Wellness Officer, Department leadership (aggregate only)
Distress by shift type: Aggregate score indexed by day shift / night shift / weekend shift. Helps staffing decisions. No names.
High-load period detector: Automatically flags weeks where distress index was >20% above baseline. Helps identify systemic drivers (surge, short staffing).
Recovery curve: How long does the aggregate score take to return to baseline after a high-demand week?
Program ROI Report
admin/dashboard/roi
Shown to: CFO, Chief HR Officer
Estimated turnover reduction: Based on peer-reviewed formula ($500K–$2M per physician replacement × estimated retention improvement from peer support engagement rate).
Peer support cost per physician per year: Bridge license cost vs. EAP usage rate (bridges are 14× more likely to be used than traditional EAP).
Medical error correlation: Available only as institutional trend after 180-day baseline. Never individual-level. Optional — must be enabled by physician council vote.
Program Health
admin/dashboard/program
Shown to: Chief Wellness Officer, Bridge Customer Success
Active user rate: % of licensed physicians who opened Bridge in last 30 days (not what they did — just that they opened it).
Circle of Trust completion rate: % who completed onboarding (built a circle). Leading indicator of engagement.
Feature adoption breakdown: Breathing sessions vs. peer outreach vs. meditations vs. check-ins. Category-level only.
Net Promoter Score: Monthly in-app pulse survey (2 questions). Aggregate NPS score displayed.

React Native (Expo SDK 51+) with TypeScript. Navigation: Expo Router (file-based). State: Zustand + React Query. Animations: Reanimated 3 + Moti. Backend Architecture — Cursor

Data models, API routes, services, and infrastructure. Build with Node.js + PostgreSQL + Redis.
Tech Stack for Backend

Node.js (v22) + TypeScript + Fastify. Database: PostgreSQL 16 (primary) + Redis 7 (cache + queues). ORM: Prisma. Auth: Auth0 + SAML 2.0 (hospital SSO). FHIR: Epic FHIR R4 via Intersystems IRIS SDK. Queue: BullMQ on Redis. Hosting: AWS (EKS for services, RDS for Postgres, ElastiCache for Redis). HIPAA-compliant deployment (Business Associate Agreement required with AWS).

Core Data Models

// physicians table — PII is minimal by design
table physicians {
  id              uuid        PRIMARY KEY DEFAULT gen_random_uuid()
  npi             varchar(10) UNIQUE NOT NULL        -- NPI registry verified
  hospital_id     uuid        REFERENCES hospitals(id)
  specialty       varchar(50)
  career_stage    varchar(20)  -- resident | attending | fellow | senior
  onboarding_done boolean     DEFAULT false
  wearable_type   varchar(20)  -- apple_watch | garmin | whoop | null
  threshold_config jsonb       -- {sustained_days: 3, sustained_score: 70, acute: 90}
  notification_prefs jsonb     -- {silent_start: "07:00", silent_end: "19:00", max_per_day: 1}
  created_at      timestamptz DEFAULT now()
  deleted_at      timestamptz  -- soft delete only, never hard delete
}

// circle_members — the trust graph
table circle_members {
  id              uuid        PRIMARY KEY
  owner_id        uuid        REFERENCES physicians(id)   -- whose circle
  member_id       uuid        REFERENCES physicians(id)   -- who is in it
  role            varchar(20) -- first_reach | backup | checkin_buddy
  notify_rule     varchar(30) -- always | delayed_4h | manual_only
  status          varchar(20) -- pending | accepted | declined | limited
  invited_at      timestamptz
  accepted_at     timestamptz
  UNIQUE (owner_id, member_id)
}

// distress_scores — computed, never raw biometrics
table distress_scores {
  id              uuid        PRIMARY KEY
  physician_id    uuid        REFERENCES physicians(id)
  date            date        NOT NULL
  composite_score smallint   -- 0–100, higher = more distress
  ehr_sub_score   smallint
  wearable_sub_score smallint -- NULL if no wearable connected
  self_report_sub_score smallint
  signal_flags    jsonb       -- {ehr_afterhours: true, hrv_low: true, ...}
  trigger_fired   boolean    DEFAULT false
  trigger_type    varchar(20) -- sustained | acute | hard_event | null
  UNIQUE (physician_id, date)
}

-- RAW biometrics from wearable NEVER stored server-side --
-- Computed only on-device via HealthKit / Health Connect --
-- Only the sub_score integer lands in the database --

// peer_messages — encrypted in transit and at rest
table peer_messages {
  id              uuid        PRIMARY KEY
  sender_id       uuid        REFERENCES physicians(id)
  recipient_id    uuid        REFERENCES physicians(id)
  message_body    text        -- AES-256 encrypted, key in physician vault
  message_type    varchar(20) -- outreach | response | schedule | flag
  status          varchar(20) -- sent | delivered | read | responded
  scheduled_at   timestamptz -- for social recovery scheduling
  created_at      timestamptz
  expires_at      timestamptz DEFAULT now() + interval '30 days'
  -- Auto-purged after 30 days. No message retention beyond this. --
}

// hospital_aggregate_snapshots — pre-computed, never queryable at individual level
table hospital_aggregate_snapshots {
  id              uuid        PRIMARY KEY
  hospital_id     uuid        REFERENCES hospitals(id)
  week_start      date
  active_physicians int
  pct_elevated    numeric(5,2) -- % above threshold. NULL if active_physicians < 10
  peer_support_utilization numeric(5,2)
  total_wellness_sessions int
  median_recovery_days numeric(5,2)
  created_at      timestamptz
  UNIQUE (hospital_id, week_start)
}

Background Services

Score Computation Service

Runs nightly (2am local hospital time). For each active physician: fetches yesterday's EHR signals from Epic, retrieves on-device-computed wearable sub-score (from last POST), applies weighting formula, writes composite score to distress_scores. Checks against sustained and acute thresholds. If trigger conditions met: enqueues notification job (BullMQ).

// BullMQ job schema
{
  jobName: "score.trigger.notify",
  data: {
    physician_id: "uuid",
    trigger_type: "sustained|acute|hard_event",
    score: 78,
    copy_variant: 3   // selects from copy library
  },
  opts: { delay: 0, attempts: 3 }
}
Epic FHIR Sync Service

Runs per-hospital on staggered schedule. Uses Epic's bulk FHIR export (Group/$export) scoped to physician login events and note timing metadata only. Parses out: login timestamps, note creation times, encounter open/close times. No patient data is stored or transmitted — only physician workflow timing.

Requires: Epic App Orchard registration, SMART on FHIR backend authorization, hospital IT approval. Scopes requested: system/AuditEvent.read, system/Practitioner.read, system/Encounter.read (metadata only).

Aggregate Snapshot Generator

Runs every Sunday 3am. Reads the past week's distress_scores for each hospital. Computes aggregate metrics. If active physician count < 10 for any sub-group, suppresses that metric (writes NULL). Writes to hospital_aggregate_snapshots. Individual-level query access to distress_scores table is blocked at the Postgres RLS (Row Level Security) policy level — only the physician's own JWT can read their own rows.

Notification Delivery Service

Consumes from BullMQ. Respects physician notification preferences (silent hours, max 1/day). Deduplicates — if physician already received a notification today, job is dropped. Delivery targets: push notification via Expo Push API (APNs + FCM). Falls back to in-app inbox if push is disabled. Tracks delivery and read status per notification.

Message Expiry Service

Runs hourly. Deletes peer messages where expires_at < now(). Messages expire 30 days after creation. Runs a VACUUM ANALYZE after deletion batch. This is a hard data retention policy — no exceptions, no recovery. The hospital cannot request message archives.

Concern Flag Resolver

Runs daily. For each active concern flag: checks if the flagged physician's own score is elevated. If yes, increases the weight of peer support suggestions in their next trigger notification. Auto-expires flags after 14 days. Tracks "flag + elevated score" convergence for reporting to Chief Wellness Officer (count only, no names).

Security & Compliance

HIPAA Technical Safeguards
  • All data at rest: AES-256 encryption (AWS KMS managed keys)
  • All data in transit: TLS 1.3 minimum
  • Peer messages: end-to-end encrypted (AES-256, key in physician HealthKit Keychain)
  • Database: PostgreSQL RLS — physicians can only read their own rows in distress_scores, circle_members, peer_messages
  • Admin JWT cannot query individual-level data tables — blocked at API middleware layer
  • BAA required with AWS, Expo, any third-party service with data access
  • Audit log for all API calls: stored 6 years (HIPAA requirement)
Auth Architecture
  • Hospital SSO: SAML 2.0 via Auth0 SAML adapter. Hospital IDP maintains identity.
  • Mobile JWT: short-lived (1h) access tokens + 30-day refresh tokens. Stored in device secure enclave (Keychain / Android Keystore).
  • Epic OAuth: physician-level SMART on FHIR. Scope limited to practitioner's own audit data.
  • Admin JWT: separate token type, role=hospital_admin. Cannot be elevated to physician permissions even with exploit — separate Postgres role.
  • MFA: required for admin accounts. Recommended (not required) for physician accounts to reduce friction.
  • Rate limiting: 100 req/min per physician JWT. 20 req/min for score submission.
